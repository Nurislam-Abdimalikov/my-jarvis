# Requirements Document

## Introduction

`portfolio-polish` — это работа по доведению проекта Jarvis (локальный голосовой
ассистент для macOS на Python) до состояния «настоящего, живого, технически
сильного indie-AI-проекта», пригодного для портфолио на GitHub и заявок в
OpenAI / OSS-программы.

Ключевое ограничение: **вся активность должна быть реальной и естественной**.
Никаких фейковых коммитов, накруток, спам-PR или искусственной активности. Все
улучшения — это настоящие артефакты (документация, код, релизы, демо) и реальные
рабочие процессы (git-ветки, коммиты, CHANGELOG).

Важный контекст платформы (исправляет частую ошибку): это **локальный desktop-демон
для macOS на Apple Silicon**, а не веб-приложение. Поэтому деплой на Vercel или
веб-хостинг **не применяется**. «Живое демо» — это скринкаст (GIF/видео) и
скриншоты, опционально статический лендинг; «деплой» означает упаковку и запуск
desktop-приложения (например, автозапуск через LaunchAgent), а не публикацию веб-сервиса.

Данный спек охватывает прежде всего **портфолио-артефакты и рабочие процессы**
(анализ, README, профиль GitHub, CHANGELOG, демо, TASKS.md, git-workflow), а
также **рекомендации по фичам и архитектуре**. Реализация самих новых фич кода
выносится в отдельные спеки и здесь только описывается и приоритизируется.

Все генерируемые тексты предоставляются как готовые к использованию markdown-блоки
и команды на русском языке (если не указано иное).

## Glossary

- **Project (Проект)**: репозиторий Jarvis — локальный голосовой ассистент для
  macOS на Python 3.11+, пакет `src/jarvis`.
- **Maintainer (Мейнтейнер)**: автор/владелец проекта, выполняющий рекомендации.
- **Analysis_Report (Отчёт-анализ)**: markdown-документ с оценкой текущего
  состояния проекта: слабые места, приоритеты, быстрые победы.
- **README**: файл `README.md` в корне репозитория.
- **Profile_Guide (Гайд по профилю)**: набор готовых текстов и рекомендаций для
  профиля GitHub: bio, описание репозитория, pinned repos, topics/tags.
- **Feature_Catalog (Каталог фич)**: структурированный список предлагаемых новых
  возможностей, сгруппированный по категориям.
- **CHANGELOG**: файл `CHANGELOG.md` в формате Keep a Changelog с версиями по
  SemVer.
- **Release_Notes (Заметки к релизу)**: текст описания GitHub Release для версии.
- **Commit_Convention (Соглашение о коммитах)**: правила формата сообщений
  коммитов (Conventional Commits) с примерами.
- **Demo_Assets (Демо-материалы)**: GIF/видео-скринкаст, скриншоты и инструкции
  по их созданию для desktop-приложения.
- **Packaging_Guide (Гайд по упаковке)**: пошаговая инструкция упаковки и запуска
  desktop-приложения на macOS (например, LaunchAgent / автозапуск), без веб-деплоя.
- **Architecture_Review (Ревью архитектуры)**: оценка текущей структуры кода и
  рекомендации по улучшению структуры папок и best practices.
- **TASKS_File (Файл задач)**: файл `TASKS.md` в корне репозитория — живая
  дорожная карта проекта в виде markdown-чеклиста с секциями TODO / IN PROGRESS / DONE.
- **Git_Workflow_Guide (Гайд по git-процессу)**: пошаговые инструкции с готовыми
  git-командами для разработки каждой новой фичи в отдельной ветке.

## Requirements

### Requirement 1: Отчёт-анализ проекта с быстрыми победами

**User Story:** Как мейнтейнер, я хочу получить честный анализ текущего состояния
проекта, чтобы понимать, что выглядит слабо и что можно улучшить за один день.

#### Acceptance Criteria

1. THE Analysis_Report SHALL описывать текущее состояние Project по разделам:
   код, документация, тесты, CI/security, презентация на GitHub.
2. THE Analysis_Report SHALL перечислять выявленные слабые места с указанием
   конкретных файлов или каталогов Project.
3. THE Analysis_Report SHALL содержать раздел «Быстрые победы» со списком задач,
   каждая из которых выполнима силами одного мейнтейнера в пределах одного
   рабочего дня.
4. WHERE рекомендуется добавить новую возможность, THE Analysis_Report SHALL
   указывать категорию возможности и затрагиваемые файлы или модули.
5. THE Analysis_Report SHALL присваивать каждому пункту улучшения приоритет из
   фиксированного набора значений: high, medium, low.
6. THE Analysis_Report SHALL основываться на фактическом содержимом репозитория и
   не предлагать действий, создающих искусственную или фейковую активность.

### Requirement 2: Улучшение README поверх существующего

**User Story:** Как мейнтейнер, я хочу усилить уже хороший русскоязычный README,
чтобы он выглядел как README зрелого OSS-проекта, не теряя текущего содержания.

#### Acceptance Criteria

1. THE README SHALL сохранять русский язык как основной язык документа.
2. THE README SHALL содержать разделы: описание, возможности, установка,
   использование, технологический стек, дорожная карта, вклад (contribution),
   лицензия.
3. THE README SHALL содержать набор бейджей статуса, включающий как минимум:
   статус CI, версию Python, лицензию.
4. THE README SHALL содержать раздел вклада (contribution) со ссылкой на
   Git_Workflow_Guide и Commit_Convention.
5. WHERE в README ссылается на Demo_Assets, THE README SHALL встраивать
   GIF-превью или скриншот в верхней части документа.
6. WHEN мейнтейнер запрашивает обновление README, THE README SHALL предоставляться
   как готовый к вставке markdown-блок.
7. THE README SHALL сохранять корректность существующих внутренних ссылок на
   `docs/SETUP.md`, `docs/SECURITY.md` и `docs/ROADMAP.md`.

### Requirement 3: Гайд по профилю GitHub и метаданным репозитория

**User Story:** Как мейнтейнер, я хочу готовые тексты и рекомендации для профиля
GitHub, чтобы проект и аккаунт выглядели профессионально для рекрутеров OpenAI/OSS.

#### Acceptance Criteria

1. THE Profile_Guide SHALL предоставлять готовый текст bio профиля GitHub в
   пределах лимита 160 символов.
2. THE Profile_Guide SHALL предоставлять готовый текст описания (description)
   репозитория в пределах лимита 350 символов.
3. THE Profile_Guide SHALL предоставлять список topics/tags репозитория,
   содержащий от 5 до 20 тегов в формате, допустимом для GitHub topics
   (строчные буквы, цифры и дефисы).
4. THE Profile_Guide SHALL предоставлять рекомендации по pinned-репозиториям с
   указанием порядка закрепления.
5. THE Profile_Guide SHALL предоставлять все тексты как готовые к копированию
   блоки.
6. THE Profile_Guide SHALL основываться на реальных характеристиках Project и не
   содержать ложных утверждений о его возможностях.

### Requirement 4: Каталог предложенных фич по категориям

**User Story:** Как мейнтейнер, я хочу структурированный список идей фич, чтобы у
проекта была убедительная, реалистичная дорожная карта развития.

#### Acceptance Criteria

1. THE Feature_Catalog SHALL содержать ровно четыре категории: общие полезные
   фичи, AI-фичи, улучшения UI, улучшения «для вида разработчика»
   (developer-experience).
2. THE Feature_Catalog SHALL предлагать как минимум три фичи в каждой из четырёх
   категорий.
3. THE Feature_Catalog SHALL для каждой фичи указывать затрагиваемые модули или
   файлы Project.
4. THE Feature_Catalog SHALL для каждой фичи присваивать оценку сложности из
   фиксированного набора значений: small, medium, large.
5. THE Feature_Catalog SHALL помечать каждую UI-фичу как соответствующую
   платформе desktop macOS (например, tray icon, overlay, notifications), а не
   веб-интерфейсу.
6. THE Feature_Catalog SHALL предлагать только фичи, реализуемые в рамках текущего
   стека Project (Python, macOS, локальные модели и OpenAI-совместимый LLM).

### Requirement 5: CHANGELOG, заметки к релизу и соглашение о коммитах

**User Story:** Как мейнтейнер, я хочу профессиональную историю изменений и
правила коммитов, чтобы проект выглядел активно и аккуратно сопровождаемым.

#### Acceptance Criteria

1. THE CHANGELOG SHALL следовать формату Keep a Changelog с заголовками версий по
   семантическому версионированию (SemVer).
2. THE CHANGELOG SHALL содержать запись `Unreleased` для незавершённых изменений.
3. THE Commit_Convention SHALL определять формат сообщений коммитов по стандарту
   Conventional Commits с перечнем допустимых типов (например, feat, fix, docs,
   refactor, test, chore).
4. THE Commit_Convention SHALL содержать как минимум пять примеров корректных
   сообщений коммитов, основанных на реальных задачах Project.
5. WHEN готовится новый релиз, THE Release_Notes SHALL предоставляться как
   готовый markdown-текст со ссылкой на соответствующую версию CHANGELOG.
6. THE CHANGELOG SHALL фиксировать только реально внесённые изменения и не
   содержать вымышленных записей.

### Requirement 6: Демо-материалы и упаковка desktop-приложения

**User Story:** Как мейнтейнер, я хочу инструкции по демо и упаковке, чтобы
показать проект «вживую» без веб-деплоя.

#### Acceptance Criteria

1. THE Demo_Assets SHALL включать инструкцию по записи скринкаста (GIF или видео),
   демонстрирующего голосовой цикл: wake word → команда → голосовой ответ.
2. THE Demo_Assets SHALL включать перечень рекомендуемых скриншотов с указанием
   демонстрируемого сценария для каждого.
3. THE Demo_Assets SHALL указывать целевой каталог хранения медиафайлов внутри
   репозитория.
4. THE Packaging_Guide SHALL описывать упаковку и запуск Project как
   desktop-приложения macOS, включая автозапуск через LaunchAgent.
5. THE Packaging_Guide SHALL НЕ включать шаги веб-деплоя на Vercel или
   веб-хостинг.
6. IF предлагается статический лендинг, THEN THE Packaging_Guide SHALL пометить
   его как опциональный и независимый от запуска desktop-приложения.
7. THE Demo_Assets SHALL содержать конкретные команды для записи экрана и
   конвертации в GIF на macOS.

### Requirement 7: Ревью архитектуры и рекомендации по структуре

**User Story:** Как мейнтейнер, я хочу оценку архитектуры с конкретными
рекомендациями, чтобы код соответствовал best practices, если есть слабые места.

#### Acceptance Criteria

1. THE Architecture_Review SHALL оценивать текущую структуру каталогов
   `src/jarvis/{core,audio,stt,tts,brain,skills}` и выявлять слабые места.
2. WHERE архитектура требует улучшения, THE Architecture_Review SHALL предлагать
   конкретную целевую структуру папок с обоснованием.
3. THE Architecture_Review SHALL перечислять применимые к Python-проекту
   best practices (например, разделение интерфейсов, типизация, обработка ошибок,
   управление конфигурацией).
4. IF существующая архитектура признана достаточной для своего масштаба, THEN THE
   Architecture_Review SHALL зафиксировать это и предложить только инкрементальные
   улучшения.
5. THE Architecture_Review SHALL сохранять обратную совместимость точки входа
   `python -m jarvis` в предлагаемых изменениях.

### Requirement 8: Живая дорожная карта в TASKS.md

**User Story:** Как мейнтейнер, я хочу поддерживаемый файл TASKS.md, чтобы вести
реальную дорожную карту и отслеживать прогресс.

#### Acceptance Criteria

1. THE TASKS_File SHALL содержать ровно три секции: TODO, IN PROGRESS, DONE.
2. THE TASKS_File SHALL представлять каждую задачу как элемент markdown-чеклиста с
   кратким описанием.
3. WHEN задача завершается, THE TASKS_File SHALL отмечать её чекбокс как
   выполненный и перемещать её в секцию DONE с датой завершения в формате
   `YYYY-MM-DD`.
4. WHEN задача завершается, THE Maintainer SHALL получать предложение следующей
   задачи из секции TODO.
5. WHILE задача находится в работе, THE TASKS_File SHALL хранить её в секции
   IN PROGRESS.
6. THE TASKS_File SHALL отражать только реальные задачи и реальный статус их
   выполнения.

### Requirement 9: Git-процесс по фиче с готовыми командами

**User Story:** Как мейнтейнер, я хочу пошаговый git-процесс для каждой новой
фичи, чтобы история репозитория выглядела чисто и профессионально.

#### Acceptance Criteria

1. WHEN начинается работа над новой фичей, THE Git_Workflow_Guide SHALL
   предоставлять команду создания отдельной ветки с именем в формате
   `feature/<kebab-case-name>`.
2. THE Git_Workflow_Guide SHALL предоставлять пошаговую последовательность команд:
   создание ветки, стейджинг, коммит, пуш с установкой upstream.
3. THE Git_Workflow_Guide SHALL предоставлять сообщения коммитов, соответствующие
   Commit_Convention.
4. THE Git_Workflow_Guide SHALL предоставлять команду создания pull request через
   CLI (`gh pr create`).
5. THE Git_Workflow_Guide SHALL предписывать пуш в ветку фичи, а не напрямую в
   `main`.
6. IF команда является разрушительной (например, `push --force`, `reset --hard`),
   THEN THE Git_Workflow_Guide SHALL помечать её предупреждением и не включать в
   стандартный поток.
7. THE Git_Workflow_Guide SHALL предоставлять все команды как готовые к выполнению
   блоки с подставленными примерными значениями.
