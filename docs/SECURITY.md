# Security — реакция на утечку API-ключа

Этот документ — рабочий playbook на случай, когда API-ключ Jarvis утёк или есть подозрение, что утёк. Читается обычно в стрессе, поэтому шаги короткие и упорядочены по приоритету.

Главное правило: **отзыв ключа в дашборде провайдера всегда идёт первым**. Всё остальное (правка `.env`, чистка истории, force-push) — после.

## Если ключ утёк локально (но не в git)

Сценарий: ключ показали на скриншоте, скинули в чат, открыли `.env` через `screen share`, оставили терминал с `cat .env` без присмотра. В git ничего не попало.

Чек-лист:

1. Сразу выставить права на `.env`, чтобы файл не читался другими пользователями системы:
   ```bash
   chmod 600 .env
   ```
2. Убедиться, что `.env` действительно не закоммичен:
   ```bash
   git log --all -- .env
   ```
   Команда должна вернуть пустой вывод. Если что-то нашлось — переключайся в раздел [«Если ключ попал в git-историю»](#если-ключ-попал-в-git-историю).
3. Если есть подозрение, что ключ кто-то реально увидел и мог сохранить — ротировать его в дашборде провайдера. Если уверен, что ключ остался только в локальном `.env` и `chmod 600` уже выставлен — ротация не обязательна.

Прямые ссылки на дашборды для ротации:

| Провайдер         | Дашборд ротации                                     |
|-------------------|-----------------------------------------------------|
| AIHubMix          | https://aihubmix.com/                               |
| Mistral           | https://console.mistral.ai/api-keys/                |
| Google AI Studio  | https://aistudio.google.com/apikey                  |
| ElevenLabs        | https://elevenlabs.io/app/settings/api-keys         |

После ротации — обновить значение в `.env`, повторно выставить `chmod 600 .env`, перезапустить ассистента.

## Если ключ попал в git-историю

Сценарий: `.env` или фрагмент с ключом всё-таки попал в коммит. Даже если коммит локальный — действуй так, будто он уже отправлен в публичный мир. Публичные репозитории индексируются GitHub'ом и сторонними сканерами секретов почти мгновенно.

Пошаговый порядок (строго в этой последовательности):

1. **Немедленно отозвать ключ в дашборде провайдера.** Это первый шаг до любой работы с git. Ссылки — в таблице выше.
2. **Выпустить новый ключ** в том же дашборде.
3. **Обновить локальный `.env`** новым значением. После правки — `chmod 600 .env`.
4. **Перезапустить ассистента**, чтобы он подхватил новый ключ.
5. **Очистить значение старого ключа из истории** через `git filter-repo`:
   ```bash
   pip install git-filter-repo
   # Заменить значение старого ключа на REDACTED во всей истории:
   echo 'OLD_KEY_VALUE==>REDACTED' > /tmp/replacements.txt
   git filter-repo --replace-text /tmp/replacements.txt
   ```
   `OLD_KEY_VALUE` — фактическое значение скомпрометированного ключа.
6. **Запушить переписанную историю**:
   ```bash
   git push --force-with-lease origin main
   ```
   Требует прав на принудительный push. `--force-with-lease` безопаснее обычного `--force`: команда не перезапишет ветку, если кто-то успел запушить туда новые коммиты.
7. **Уведомить co-maintainer-ов** и попросить их сделать свежий клон репозитория или ребейз своих локальных веток. Их старые клоны после `force-push` будут расходиться с remote.

> ⚠️ Переписывание истории — это последний шаг, не первый. Сам факт того, что ключ оказался в публичном репозитории даже на минуту, означает, что его уже мог проиндексировать GitHub или сторонний сканер секретов. Поэтому отзыв ключа всегда первичен; чистка истории — косметика поверх.

## Проверка по логам провайдера

После ротации стоит проверить, что старый ключ никем не использовался в окне утечки.

Где смотреть:

- **AIHubMix** — раздел статистики/использования в дашборде https://aihubmix.com/.
- **Mistral** — usage/billing в https://console.mistral.ai/api-keys/.
- **Google AI Studio** — раздел API usage в https://aistudio.google.com/apikey (или Google Cloud Console для проектов с привязанным billing).
- **ElevenLabs** — usage history в https://elevenlabs.io/app/settings/api-keys.

На что обращать внимание:

- Всплески запросов в нерабочее время или сразу после момента предполагаемой утечки.
- Запросы с неизвестных IP или из географии, не совпадающей с твоей.
- Резкое приближение к квоте/лимиту биллинга.
- Использование моделей или эндпоинтов, которые ассистент не вызывает.

Если что-то из этого видно — фиксируй детали (timestamp, IP, объём) и связывайся с support провайдера.

## Чувствительные файлы

Файлы, которые **никогда не передаём публично**:

- `.env` — содержит реальные API-ключи. Не передавать через мессенджеры, email или облачные диски без шифрования. Для шеринга секретов между машинами — использовать менеджер паролей или зашифрованный канал.
- `logs/jarvis.log` — пишется через `loguru` и может содержать фрагменты пользовательских запросов и LLM-ответов. Перед тем как приложить лог к bug-репорту или скинуть коллеге — открыть файл и вручную вычистить любой текст, похожий на персональные данные или ключи.

Если очень нужно поделиться логом — отредактировать копию (не оригинал), убедиться, что в ней нет ничего лишнего, и приложить отредактированную версию.

## Профилактика

Чек-лист, который держит репозиторий в зелёной зоне:

- [ ] `chmod 600 .env` — после каждой правки `.env`. `scripts/setup.sh` делает это автоматически.
- [ ] `pre-commit install` выполнен один раз после клонирования. Хук блокирует коммит секретов локально.
- [ ] `bash scripts/check_env.sh` проходит зелёным — все четыре проверки `✅`.
- [ ] `pre-commit run --all-files` запущен перед каждым PR — это та же проверка, что и в CI, но локально и быстрее.

Подробности по установке pre-commit и описание `check_env.sh` — в [`docs/SETUP.md`](./SETUP.md).
